\section{Benchmark Design and Creation}\label{sec:design}

\subsection{Rule Derivation Process}\label{sec:rule-derivation}

Each benchmark requires a domain-specific rule set in DRL, derived through a semi-automated process (Fig.~\ref{fig:derivation}) comprising three stages.

\textbf{Stage~1: Source identification and LLM-assisted generation.}
For each domain, authoritative public guidelines and a corresponding real-time event stream were identified (Table~\ref{tab:sources}).
These guidelines were provided as structured context to a large language model, which produced an initial DRL rule set specifying the target fact model, CEP annotations (\texttt{@Role(Event)}, \texttt{@Expires}), and chaining behaviour.

\textbf{Stage~2a: Human refinement.}
Generated rules were manually corrected: temporal operators were aligned with domain semantics, numeric thresholds adjusted to guideline values, chaining paths restructured into multi-level derivation pipelines, and deduplication/retraction logic added for realistic WM management.

\textbf{Stage~2b: Validation.}
After each refinement cycle, the rule set was replayed against pre-collected data dumps using the characterisation collector.
Validation criteria included: chaining depth matching the expected pipeline topology, rule coverage $>$60\%, temporal operators present across categories, and stable memory under 400K+ events.
Failure on any criterion returned the process to Stage~2a.

\subsection{Benchmark Domains}\label{sec:domains}

Each domain exercises a distinct region of the CEP engine's capability space.
Fig.~\ref{fig:taxonomy} presents the processing taxonomy for each rule set.

\textbf{Aviation surveillance.}
Models an STCA system based on EUROCONTROL guidelines~\cite{eurocontrol_stca}.
ADS-B data is refined through quality filtering, kinematic anomaly detection, grid-based pair generation, CPA computation, severity classification with persistence-based hysteresis, and an alerting policy layer.
Stress profile: \emph{deep forward chaining} --- a single event triggers derived-fact cascades across all six phases.

\textbf{Financial market risk.}
Implements a MiFID~II/RTS~6-compliant market-data risk engine for crypto-asset futures~\cite{esma_mifid2_art17, eu_rts6_2017}, ingesting order-book depth, trades, mark-price, and liquidation streams.
Sliding-window rules detect trade-rate bursts and liquidation cascades.
Stress profile: \emph{high-frequency state management} --- derived analytics are singleton facts per symbol, generating intensive WM updates under bursty arrival.

\textbf{Collaborative editing moderation.}
Models a real-time anti-vandalism and bot-governance pipeline for Wikipedia, based on the ORES framework~\cite{halfaker2020ores}.
Five parallel pipelines (vandalism, bot, content, minor-edit, discussion) each progress through detection, analysis, verification, enforcement, and output stages.
Stress profile: \emph{join-intensive multi-pattern evaluation} --- vandalism detection requires cross-product joins across deletion-size tiers, user reputation, and repeat-offender history; a correlation layer joins outputs from multiple pipelines.

\subsection{Benchmarking Infrastructure}\label{sec:infra}

Each benchmark is driven by a replay engine that feeds a pre-collected event dump into a streaming-mode rule session with a controllable pseudo-clock, advanced to each event's original timestamp to preserve real-world inter-arrival patterns.
Benchmarks are implemented as JMH microbenchmarks within the Apache KIE \texttt{incubator-kie-benchmarks} repository, replaying the full event sequence to measure end-to-end throughput under controlled warm-up and iteration parameters.
